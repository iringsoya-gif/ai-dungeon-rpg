## 슬라이드 03 — 프로젝트 개요 & 동기 (1:00 ~ 2:10, 70초)

여러분, ChatGPT로 RPG를 해보신 적 있으신가요?

직접 해보면 이런 문제가 생깁니다.
매번 "내 캐릭터 HP는 80이고, 인벤토리에 단검이 있고, 현재 던전 3층에 있다"
이걸 일일이 설명해줘야 합니다.
새로고침하면 다 사라지고, 대화가 길어지면 앞 내용을 잊어버립니다.

**"AI가 진짜 GM 역할을 할 수 있을까?"**

이 질문에서 시작했습니다.

단순히 프롬프트를 넣고 답 받는 것을 넘어서,
캐릭터 상태가 자동으로 추적되고,
플레이어의 행동에 세계가 실제로 반응하며,
언제든지 돌아와서 이어할 수 있는 진짜 RPG 서비스를 만들고 싶었습니다.

그리고 이 과정에서 수업에서 배운 API 활용, 웹 개발 기술들을
하나의 완성된 서비스로 통합하는 것이 목표였습니다.

## 슬라이드 04 — 시스템 아키텍처 (2:10 ~ 3:20, 70초)

기술 스택을 크게 세 영역으로 나눌 수 있습니다.

**프론트엔드**는 React 18로 만든 SPA로 Vercel에 배포했습니다.
Zustand로 게임 상태를 관리하고,
실시간 스트리밍은 SSE, BGM은 Web Audio API로 구현했습니다.

**백엔드**는 Python FastAPI 서버로 Railway에 Docker 컨테이너로 배포했습니다.
SQLAlchemy로 SQLite 데이터베이스를 다루고,
Google OAuth로 로그인, Polar.sh로 결제를 처리합니다.

**AI 영역**은 Groq API를 통해 Meta의 LLaMA 3.3 70B 모델을 사용합니다.
GM 서사 생성과 오프닝 텍스트 생성 모두 동일 모델을 씁니다.
처음엔 오프닝에 8B 모델을 썼는데, 시점 혼용과 직업 무시 문제로 70B로 통일했습니다.

중요한 점은 Groq API가 **무료** 티어를 제공해서,
별도 비용 없이 고성능 70B 모델을 사용할 수 있었습니다.

## 슬라이드 05 — 핵심 기술: AI 게임마스터 (3:20 ~ 4:30, 70초)

이 서비스의 핵심은 AI GM을 어떻게 설계했느냐입니다.

단순히 "RPG 진행해줘"라고 하면 모델은 들쭉날쭉한 결과를 냅니다.
저는 **시스템 프롬프트를 GM의 '규칙서'로** 설계했습니다.

프롬프트는 크게 네 부분으로 구성됩니다.

첫째, **언어 규칙**: 순수 한국어만 사용하도록 명시합니다.
실제로 LLaMA 모델이 韓국어 서술 도중 일본어 한자를 섞어 쓰는 문제가 있었는데,
틀린 예시와 올바른 예시를 나란히 제시해 이를 방지했습니다.

둘째, **장면 연속성 규칙**: 직전 GM 서술의 장소에서 이야기가 이어지도록 강제합니다.
이를 어기면 마법학교 방에서 시작했는데 갑자기 시장 거리로 점프하는 문제가 생겼습니다.

셋째, **웹소설 문체 규칙**: 짧고 리드미컬한 문장으로 한국 웹소설 느낌을 냅니다.
나쁜 예와 좋은 예를 함께 제시하는 Few-shot 기법을 사용했습니다.

넷째, **구조화된 출력**: GM 응답 마지막에 반드시 JSON 블록이 오도록 형식을 지정합니다.
이 JSON에서 HP 변화, 획득 아이템, 퀘스트, 전투 상태를 추출해 서버가 자동으로 처리합니다.

## 슬라이드 06 — 핵심 기술: SSE & 컨텍스트 관리 (4:30 ~ 5:30, 60초)

**실시간 스트리밍**은 SSE(Server-Sent Events)로 구현했습니다.

플레이어가 행동을 입력하면,
FastAPI가 Groq API에 스트리밍 요청을 보내고,
청크 단위로 받은 텍스트를 SSE 이벤트로 프론트에 즉시 전달합니다.
ChatGPT처럼 글자가 타이핑되듯 나타나는 효과입니다.

스트리밍이 끝나면 'done' 이벤트로 상태 변화 JSON을 전달하고,
화면의 캐릭터 패널이 실시간으로 업데이트됩니다.

**컨텍스트 관리**도 중요한 부분입니다.
게임이 길어지면 전체 대화 기록이 모델의 컨텍스트 윈도우를 초과합니다.
8,000 토큰을 넘으면 오래된 기록을 LLaMA 8B로 요약하고,
최근 10턴 + 요약만 모델에 전달하는 슬라이딩 윈도우 방식을 씁니다.

그리고 오프닝 장면 묘사는 `[게임 시작]` 마커를 앞에 붙여서
항상 모델이 첫 장면의 설정을 인식하도록 했습니다.

## 슬라이드 07 — 주요 기능 ① (5:30 ~ 6:20, 50초)

게임 생성 단계에서 **세계관 템플릿** 다섯 종류를 고를 수 있습니다.
판타지, SF, 공포, 현대, 그리고 직접 입력하는 커스텀 모드입니다.

**직업 시스템**은 세계관에 따라 다른 직업이 제시됩니다.
판타지면 전사·마법사·도적, SF면 해커·엔지니어 같은 식으로요.
20종 프리셋 외에 "강령술사", "우주해적"처럼 직접 입력도 됩니다.

각 직업마다 HP, MP, 근력, 지능, 민첩, 카리스마 스탯이 다르게 설정됩니다.
GM은 이 스탯을 반영해서 전사는 압도적인 힘으로, 마법사는 허점을 꿰뚫는 판단으로 서술합니다.

레벨업 시에는 전 스탯이 자동으로 증가하고, 다음 레벨까지 필요 경험치는 1.5배씩 늘어납니다.

## 슬라이드 08 — 주요 기능 ② (6:20 ~ 7:00, 40초)

UI는 caveduck.io 같은 AI 채팅 서비스를 참고해서
다크 판타지 테마로 디자인했습니다.

`**행동**` 형식으로 행동을 지시하고, 일반 텍스트는 대사로 처리됩니다.
이 사실을 화면 하단에 힌트로 표시해뒀습니다.

BGM은 Web Audio API로 실시간 생성합니다.
탐험 중엔 A단조 앰비언트, 전투가 시작되면 E단조로 자동 전환됩니다.
외부 음원 파일 없이 브라우저 안에서 모두 처리합니다.

Google 계정으로 로그인하고, 무료 플랜은 활성 게임 1개,
Polar.sh를 통해 9달러 일회결제로 무제한 이용이 가능합니다.

## 슬라이드 09 — 수업 내용 활용 (7:00 ~ 8:00, 60초)

이 프로젝트가 수업 내용과 어떻게 연결되는지 정리해봤습니다.

**LLM과 프롬프트 엔지니어링** 관점에서,
모델이 토큰 단위로 텍스트를 생성한다는 원리를 이해하고,
시스템 프롬프트로 출력 형식과 문체를 제어하는 것을 직접 구현했습니다.
Few-shot 기법, 즉 좋은 예/나쁜 예를 제시해 모델의 행동을 유도하는 방법도 활용했습니다.

**API와 비동기 처리** 관점에서,
5주차에 배운 FastAPI로 백엔드 전체를 구축했고,
async/await를 활용한 비동기 스트리밍을 구현했습니다.

**데이터 구조와 상태 관리** 관점에서,
캐릭터 상태를 JSON으로 직렬화해 DB에 저장하는 방식은
데이터를 구조화해서 처리하는 수업 내용과 직결됩니다.
컨텍스트 윈도우 관리는 사실 일종의 슬라이딩 윈도우 알고리즘입니다.

**NLP 관점에서는**, 텍스트 생성 방향을 프롬프트로 제어하는 것,
즉 스타일 전이(Style Transfer)를 프롬프트 엔지니어링으로 구현한 셈입니다.

## 슬라이드 10 — 기존 AI 챗봇과의 비교 (8:00 ~ 8:50, 50초)

표로 정리해봤습니다.

가장 큰 차이는 **상태의 지속성**입니다.
ChatGPT는 사용자가 직접 "내 HP는 이제 50이야"라고 알려줘야 하지만,
이 서비스는 GM 응답에서 상태 변화를 자동으로 파싱해서 DB에 저장합니다.

**장기 플레이 맥락**도 다릅니다.
일반 챗봇은 컨텍스트 윈도우를 넘으면 앞 내용을 잊지만,
자동 요약 압축으로 이를 해결했습니다.

**하드코어 모드**처럼 게임 규칙을 시스템 수준에서 보장하는 것도 차별점입니다.
챗봇에서는 플레이어와 모델이 합의해야 하지만,
여기서는 서버가 HP=0을 감지하면 게임을 영구 종료시킵니다.

그리고 장면 반응 BGM처럼 게임 상태와 연동되는 부가 기능도 일반 챗봇에는 없는 부분입니다.

## 슬라이드 11 — 기술적 도전과 해결 (8:50 ~ 9:50, 60초)

개발 과정에서 여섯 가지 큰 문제를 만났습니다. 슬라이드에 모두 정리해뒀는데, 핵심만 말씀드리겠습니다.

**첫째, GM이 응답 없이 침묵하는 문제.**
동기 클라이언트를 비동기 컨텍스트에서 써서 이벤트 루프가 블로킹됐습니다.
AsyncGroq로 전환하고 메시지 배열 형식을 수정해서 해결했습니다.

**둘째, 일본어 한자 혼입 문제.**
LLaMA 모델이 한국어 서술 도중 微笑, こちらを指示 같은 문자를 섞어 씁니다.
프롬프트 최상단에 틀린 예시와 올바른 예시를 나란히 제시해서 해결했습니다.

**셋째, 장면 연속성 단절 문제.**
오프닝 장면이 context_manager에서 삭제되어 모델에게 전달되지 않았습니다.
삭제 대신 `[게임 시작]` 마커를 삽입하는 방식으로 수정했습니다.

**넷째, 오프닝 서술 품질 문제.**
8B 모델이 시점을 혼용하고 직업을 무시했습니다.
70B 모델로 통일하고 프롬프트에 2인칭 규칙, 직업 힌트, 3단계 구성을 명시했습니다.

## 예상 질문 & 답변

**Q1. Groq API 무료 티어의 한계는?**
분당 요청 수 제한(RPM)과 일일 토큰 한도가 있습니다. 상용 서비스라면 유료로 전환하거나 자체 모델을 운영해야 합니다. 소규모 데모/과제 수준에선 충분합니다.

**Q2. SQLite를 쓰는 이유는? 실제 서비스에선 문제 없나?**
Railway 무료 티어에서 외부 DB 비용을 줄이기 위해 SQLite를 선택했습니다. 동시 접속이 적을 때는 충분하지만, 트래픽이 늘면 PostgreSQL로 마이그레이션이 필요합니다. SQLAlchemy ORM을 쓰기 때문에 URL만 바꾸면 됩니다.

**Q3. BGM이 실제 음악처럼 들리나?**
Web Audio API 오실레이터 기반이라 멜로디가 있는 음악은 아니고, 화음 드론 형태의 앰비언트 사운드입니다. 실제 음악 파일을 쓰고 싶다면 `TRACKS` 객체에 MP3 URL을 넣으면 바로 전환됩니다.

**Q4. 컨텍스트 압축 시 중요한 정보를 잃지 않나?**
요약 과정에서 정보 손실 가능성은 있습니다. 중요 정보(캐릭터 상태, 위치)는 character_json과 시스템 프롬프트에 항상 포함되므로 핵심 상태는 보존됩니다. 서사적 세부 내용은 일부 손실될 수 있습니다.

**Q5. 하드코어 모드에서 GM이 규칙을 어길 수 있지 않나?**
GM이 `game_over: false`를 반환해도, 서버가 캐릭터 HP를 확인해서 0이면 강제로 `game.status = "dead"`로 처리합니다. 서버 로직이 GM 판단보다 우선합니다.
